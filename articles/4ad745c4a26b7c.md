## きっかけ

使用頻度が低いDBをサーバレスにすれば安くなるはず——そう思って試算したら、逆に高くなった。

担当しているサービスで DocumentDB を運用している。すでに最小スペック（db.t4g.medium）で稼働していたが、監視データを眺めていてあることに気づいた。

- CPU使用率：平均は低いが、最大は平均の数倍に達するスパイクが存在
- 接続数：ゼロになる時間帯がある
- 稼働パターン：**土日はほぼ使われず、平日も大きなスパイクが1日1回程度**

「これだけ使われていない時間があるなら、サーバレスにすればコストを削減できるのでは？」

そう考えて調査を始めた。

---

## DBのサーバレスとは？

調査を進める前に、サーバレスの概念を整理した。

「サーバレス＝完全従量課金で、使わない時間はコストゼロ」というイメージを持っていたが、それは **Lambda（FaaS）の話**であって、DBのサーバレスは少し異なる。

| | Lambda（FaaS） | DB Serverless |
| --- | --- | --- |
| アイドル時の課金 | **完全ゼロ** | **最小容量の維持費あり** |
| コールドスタート | ミリ秒オーダー | 秒〜数十秒オーダー |
| 状態管理 | ステートレス | ステートフル（接続・キャッシュを維持） |
| スケール単位 | 関数単位 | 容量単位（ACU / DCU など） |

DBがステートフルである以上、接続の維持やキャッシュの管理が必要だ。そのため、完全にゼロにはならず**最小容量の維持費**が常にかかる設計になっている。

Lambda のように「使った分だけ」という発想は、DBのサーバレスには当てはまらない。

---

## DocumentDB Serverlessの仕組み

DocumentDB Serverless では **DCU（DocumentDB Capacity Unit）** が課金単位だ。

ポイントは以下の2点だ。

**最小DCUが維持費の正体**
DCU には最小値の設定があり、アイドル状態でもこの最小DCU分のコストが発生し続ける。「使っていないのにお金がかかる」の正体はこれだ。

**コールドスタートのトレードオフ**
スケールダウン後に再びリクエストが来た際、バッファキャッシュが失われた状態から再起動するため、最初のリクエストでレイテンシが上がりやすい。これはインスタンス型にはないトレードオフだ。

---

## コスト試算してみた

現行インスタンスのコストを100として、条件別に試算した。

| 構成 | 月コスト（相対値） |
| --- | --- |
| 現行インスタンス（db.t4g.medium） | 100 |
| サーバレス（現行スペック維持で移行） | 約130 |
| サーバレス（最小DCU = 0.5） | 約60 |
| 現行インスタンス ＋ Savings Plan | 約65 |
| サーバレス（最小DCU = 0.5）＋ Savings Plan | 約40 |

### 「安全移行」の条件だと逆転する

最初に試算したのは「スペックを変えずにそのままサーバレスへ移行」するケースだ。この条件だと**サーバレスの方が高くなった。**

理由は最小DCUの維持費だ。現行スペックと同等の性能を保証しようとすると最小DCUを高めに設定せざるを得ず、アイドル時のコストが積み上がる。

今回の使用パターン（土日ゼロ、平日スパイク1回）では、アイドル時間が長いほどこのコストが響く。

### 最小DCUを下げるとサーバレスが有利になる

最小DCU = 0.5 まで下げると一気にコストが下がり、さらに Savings Plan を適用すると最も安い選択肢になる。

ただし **最小DCU = 0.5 で実際のワークロードが捌けるかどうかは未検証**のため、現時点では確定した選択肢にはできていない。

---

## 実際に移行してみた

コスト試算と並行して、実際にサーバレスへの移行を試してみた。

事前に「スナップショットの取得や複雑な手順が必要では？」と思っていたが、[AWS公式の移行手順](https://docs.aws.amazon.com/ja_jp/documentdb/latest/developerguide/docdb-serverless-migrating.html)を見ると、既存インスタンスをそのままサーバレスに変換できる形になっている。

実際にやってみた結果：

- スナップショット不要
- 数分のダウンタイムのみで切り替わった
- データ損失なし

**移行自体のハードルは思っていた数倍低かった。** コストさえ合えば移行を躊躇う理由はほぼないと感じた。

---

## 最終判断

試算と検討の結果、今回は **Savings Plan ＋ 現行インスタンス** を選択した。

最小DCU = 0.5 でのサーバレスが最もコストが安い可能性はあるが、その条件で実際のスパイクを捌けるかどうかが未検証だ。まずは Savings Plan で現行コストを削減しつつ、サーバレスの検証を進める方針とした。

### サーバレスが向くケース・向かないケース

| サーバレスが向くケース | インスタンスが向くケース |
| --- | --- |
| 完全にゼロの時間が長い | 最小DCUを高く設定せざるを得ない |
| 最小DCUを低く抑えられる | 予測可能で継続的なワークロード |
| スパイクが不規則で予測困難 | ある程度安定した使用量がある |

---

## まとめ

**「あまり使わないDBをサーバレスにすれば安くなる」は必ずしも正しくない。**

DBのサーバレスには最小容量の維持費が存在するため、使わない時間が長くても一定のコストがかかり続ける。コスト削減効果が出るかどうかは、**最小DCUをどこまで下げられるか**と**実際の使用パターン**のかけ合わせによって変わる。

今回の調査で気づいたのは「サーバレス＝安い」という前提を持ちすぎていたことだ。移行を検討する際は、使用パターンを実測した上で条件別に試算することが重要だと感じた。

同じようにDBのサーバレス移行を検討している方の参考になれば幸甚です。
